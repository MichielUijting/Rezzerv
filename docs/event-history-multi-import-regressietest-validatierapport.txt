Rezzerv v01.07.12 validatierapport

Doel van deze release
- Verifiëren dat winkelimport voor hetzelfde artikel meerdere losse inventory-events bewaart.
- Admin-regressietest uitbreiden met een scenario waarin hetzelfde artikel meerdere keren wordt opgeboekt.
- Historieweergave stabieler maken voor meerdere events met hetzelfde tijdstip.

Uitgevoerde wijzigingen
1. Admin browser-regressietest uitgebreid
- Nieuw scenario toegevoegd: "Winkelimport bewaart twee losse events voor hetzelfde artikel en Historie toont beide".
- Scenario verwerkt Tomaten via Jumbo twee extra keer achter elkaar.
- Test controleert daarna:
  - voorraad stijgt na elke verwerking;
  - /api/dev/article-history bevat minimaal 3 store_import-events voor Tomaten via Jumbo;
  - Artikeldetails -> Historie toont minimaal 3 historiekaarten;
  - Historie toont minimaal 3 keer "Geïmporteerd via Jumbo".

2. Historie-rendering gestabiliseerd
- Live history mapping geeft nu event-id mee aan de frontend.
- ArticleHistoryTab gebruikt event-id als React key wanneer beschikbaar.
- Hierdoor blijven meerdere events met hetzelfde timestamp-seconde stabiel en individueel renderen.

Verificatie uitgevoerd
- Python compile-check backend/app/main.py: geslaagd.
- Frontend build via Vite: geslaagd.
- Gerichte API-validatie uitgevoerd met drie Jumbo-opboekingen voor Tomaten:
  - meerdere losse inventory_events opgeslagen;
  - article-history retourneert minimaal 3 rows;
  - rows hebben unieke event-id's.

Niet gewijzigd
- Geen brede UI-ombouw.
- Geen wijziging aan voorraadmodel.
- Geen nieuwe provider toegevoegd.
- Geen wijziging aan Lidl-flow behalve behoud binnen regressietestset.
