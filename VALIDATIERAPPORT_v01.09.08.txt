Rezzerv validatierapport
Versie: 01.09.08
Release: Kassa Release 1B - backend receipt import + parsing + receipt API
Datum: 2026-03-19

Samenvatting
- Doel van deze release: Kassabon -> Tabel realiseren zonder voorraadwrites en zonder UI-wijziging.
- Oordeel: gedeeltelijk GO voor de backendfeature zelf; geen UI-wijziging uitgevoerd; geen inventory_events aangemaakt door de nieuwe receipt-routes.
- Belangrijke nuance: docker compose build --no-cache kon in deze omgeving niet worden uitgevoerd omdat Docker niet beschikbaar was.

Wat is gewijzigd
1. Nieuwe backend receipt-service toegevoegd:
   - backend/app/services/receipt_service.py
   Functies:
   - default receipt sources aanmaken voor local_folder en scan_folder
   - raw receipt ingest
   - sha256 dedupe op household + bestand
   - PDF tekstlaag parsing via pypdf
   - pragmatische bitmap-route met review_needed wanneer OCR niet beschikbaar is
   - receipt_table en receipt_table_lines persistente opslag
   - reparse van bestaande raw receipt
   - source scan met logging in receipt_processing_runs

2. Nieuwe backend API-endpoints toegevoegd in backend/app/main.py:
   - POST /api/receipts/import
   - POST /api/receipts/source-scan
   - GET /api/receipts?householdId={id}
   - GET /api/receipts/{receiptTableId}
   - POST /api/receipts/{receiptTableId}/reparse

3. Backend dependencies uitgebreid:
   - pypdf==5.4.0
   - python-multipart==0.0.20

4. Kleine baseline-stabilisatie meegenomen:
   - seed insert voor inventory vult nu expliciet status='active' zodat een schone sqlite-start niet faalt op NOT NULL inventory.status.

5. Versienummering consistent opgehoogd naar 01.09.08:
   - VERSION.txt
   - version.json
   - frontend/version.json
   - frontend/public/version.json
   - frontend/dist/version.json
   - frontend/package.json
   - frontend/package-lock.json
   - frontend productiebuild opnieuw opgebouwd

Wat is niet gewijzigd
- Geen writes naar inventory_events vanuit de nieuwe receipt-flow
- Geen Kassa-button koppeling
- Geen routerwijziging
- Geen redesign
- Geen wijziging aan bestaande artikeldetail-endpoints
- Geen wijziging aan bestaande voorraadprojectie

Wat is getest
1. Python compile-check:
   - python -m compileall -q backend/app
   Resultaat: geslaagd

2. Frontend productiebuild:
   - npm run build
   Resultaat: geslaagd

3. API-validatie via FastAPI TestClient tegen een schone sqlite testdatabase met aparte receipt storage root:
   - GET /api/health -> 200 {"status":"ok"}
   - POST /api/auth/login -> 200
   - POST /api/receipts/import met PDF bon -> 201, receipt_table aangemaakt, parse_status=parsed
   - POST /api/receipts/import met duplicate PDF -> 200, duplicate=true
   - POST /api/receipts/import met PNG bon -> 201, receipt_table aangemaakt, parse_status=review_needed
   - POST /api/receipts/import met niet-bon PDF -> 201, raw_receipt opgeslagen, receipt_table_id=null, parse_status=failed
   - GET /api/receipts -> lijst werkt
   - GET /api/receipts/{id} -> detail werkt
   - POST /api/receipts/{id}/reparse -> werkt
   - POST /api/receipts/source-scan -> verwerkt 1 file uit local_folder
   - Controle inventory_events voor/na receipt-tests -> onveranderd (5 -> 5)

4. Samenvatting van de backend testcase:
   - raw_receipts: 4
   - receipt_tables: 3
   - receipt_table_lines: 3
   - receipt_processing_runs: 1

Wat is niet getest
- docker compose build --no-cache (Docker niet beschikbaar in deze omgeving)
- runtime containers in Docker
- echte OCR op bitmapbestanden met een OCR-engine zoals Tesseract (niet toegevoegd in deze release)
- UI-koppeling via Kassa-button (bewust buiten scope)
- handmatige bronconfiguratie in UI (bewust buiten scope)

Bronnen en testopmerking voor local/scan folders
- Deze release seedt standaard receipt sources voor household 1:
  - 1-local-folder
  - 1-scan-folder
- In de testomgeving lagen deze folders onder de receipt data root /sources/1/...
- De source-scan endpoint verwacht een bestaand source_id.

Risiconiveau
- Middel

Toelichting risico
- Laag risico op bestaande voorraadflow omdat inventory_events niet wordt aangeraakt.
- Middel risico in parserkwaliteit, vooral voor bitmapbestanden omdat deze release nog geen volwaardige OCR-engine integreert.
- Laag tot middel risico op startup, omdat de release ook een baseline-fix bevat voor inventory.status in een schone sqlite-start.

Conclusie
- De gevraagde backendstap Kassabon -> Tabel is functioneel toegevoegd.
- PDF met tekstlaag parseert nu naar receipt_table + receipt_table_lines.
- Bitmapbestanden worden opgeslagen en krijgen een receipt_table met review_needed als OCR niet beschikbaar is.
- Geen voorraadwrites aangetroffen.
- De volgende logische stap is Release 1C: Kassa-button koppelen aan de lijst van receipt tables.
